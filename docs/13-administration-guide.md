# Administration Guide

## Purpose

Use this page to manage the administrative work needed to keep BRIMS usable, governed, and correctly configured.

## Who Should Use This Page

- System administrators
- Selected project administrators

## Before You Begin

Before you start, make sure you have administrative access and understand the governance rules that apply to your environment.

## Administrative Responsibilities

Use administrative controls to keep BRIMS reliable, secure, and ready for project use.

This includes managing access, checking configuration, supporting new projects, and monitoring recurring operational issues.

## User Management

User management includes creating accounts, reviewing account details, updating access where needed, and deactivating accounts that should no longer be used.

Review user access regularly so that project membership and permissions remain appropriate.

## Role and Permission Management

Administrators are responsible for making sure roles and permissions match real working responsibilities.

System roles should be kept limited to users who need wider administrative access.

Project roles should support day-to-day work without giving unnecessary permissions.

## Reference Data and Configuration

Reference data and configuration should be reviewed whenever new projects, workflows, or study requirements are introduced.

Keep shared values and configuration settings consistent so that data entry and reporting remain reliable across the system.

## Preparing for New Projects

Before onboarding a new project, confirm that the required people, permissions, and supporting configuration are in place.

This includes checking user access, project setup readiness, and any dependencies such as REDCap-linked workflows.

## Operational Monitoring

Operational monitoring includes reviewing repeated user issues, checking for incorrect access, and confirming that configuration remains aligned with current practice.

When the same problem appears more than once, review whether it is caused by permissions, process gaps, or outdated configuration.
